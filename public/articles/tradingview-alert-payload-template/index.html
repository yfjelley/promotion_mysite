TradingView Alert Payload Template

A TradingView alert payload should be designed like an interface contract. If the payload only says buy or sell, the receiver cannot reliably prevent duplicates, route orders, apply risk checks or explain what happened later.

Technical engineeringNo custodySource handoffRemote collaboration

A useful TradingView alert payload template includes strategy, version, symbol, action, size model, signal time, event ID inputs, endpoint authentication requirements, risk profile and audit fields.

Fields that make a signal auditable

The receiver needs enough context to understand one trading intent. Include strategy name, strategy version, symbol, action, side, size model, timeframe, signal time, bar time and a customer-visible risk profile. These fields allow the system to produce an event log that a client can review.

  • Use explicit actions such as enter_long, exit_long, reduce_only or cancel.
  • Include strategy_version when the Pine Script or rule set changes.
  • Keep account identifiers and secrets out of public examples.

Fields that prevent duplicate execution

The payload should provide stable inputs for an event ID. A common pattern combines strategy, symbol, action, timeframe and bar time, then stores the result before order routing. The system can then treat repeated delivery as a duplicate event instead of a new order.

  • Do not generate a random ID inside every retry.
  • Group cooldown rules by strategy, symbol and action.
  • Log duplicate events with a duplicate_signal reason.

Fields that support risk review

Risk controls need visible context. The payload should allow the receiver to compare intended size, current position, max position, reduce-only status, price protection and manual pause state before any third-party API call is made.

  • Use size_model instead of hard-coding every quantity in the service.
  • Record risk_profile and rule version in logs.
  • Never make profit results the software acceptance criterion.

Payload fields to include

  • strategy, strategy_version, symbol, timeframe and signal_time are present.
  • action, side and size_model are explicit enough for order-intent generation.
  • bar_time or equivalent event ID input remains stable across repeated delivery.
  • No reusable credential is placed in the payload body; endpoint tokens are revocable, rotatable and redacted from logs.
  • risk_profile, customer note and alert version are visible in audit logs.
