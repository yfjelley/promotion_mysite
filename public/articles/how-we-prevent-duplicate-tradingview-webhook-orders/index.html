How We Prevent Duplicate TradingView Webhook Orders

Duplicate TradingView webhook orders are usually not solved by one setting. A production-ready workflow needs stable event IDs, cooldown windows, risk decisions, logging and a replayable acceptance test.

Technical engineeringNo custodySource handoffRemote collaboration

Preventing duplicate TradingView webhook orders requires stable event IDs, strategy-symbol-action grouping, cooldown windows, idempotent order routing, risk reject logs and replay tests.

Start with a payload that can identify one event

The webhook payload should contain only the non-sensitive fields needed to describe one trading intent: strategy, version, symbol, action, size model, timeframe, signal time or bar time. Do not place account passwords, API keys, reusable authentication secrets or other sensitive values in the webhook body.

  • Do not rely on a plain buy or sell string.
  • Include enough stable fields to distinguish strategy, symbol, direction and time.
  • Authenticate with HTTPS, a revocable high-entropy endpoint token, TradingView's published sender IP allowlist, rate limits and strict schema validation; redact and rotate the token.

Apply idempotency before order routing

The receiver should check whether the event ID has already been processed before any order is prepared. A second layer checks the same strategy, symbol and action inside a cooldown window so repeated alerts do not become repeated orders.

  • Store every accepted, duplicated and rejected event.
  • Group cooldowns by strategy and symbol, not only globally.
  • Log duplicate events as a normal audit result instead of silently dropping them.

Treat risk rejects as successful controls

Max position, max order size, price protection, trading hours, manual pause and reduce-only rules can all reject a valid-looking webhook. That reject should be visible to the client with a risk reason and alert status.

  • Keep risk_decision and risk_reason in the event log.
  • Separate duplicate_signal, risk_rejected, api_error and manual_pause alerts.
  • Replay sample payloads before connecting real order permissions.

Duplicate-order acceptance checks

  • Sending the same payload twice produces only one order intent.
  • An invalid or rotated endpoint token is rejected without logging the token value.
  • A max-position breach is rejected with a visible risk reason.
  • Manual pause blocks new order routing before the API call.
  • The client can review event ID, duplicate status, risk decision and alert delivery.
