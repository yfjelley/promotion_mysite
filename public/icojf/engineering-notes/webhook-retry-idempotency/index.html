Engineering Note

Webhook Retry and Idempotency

A webhook endpoint should be designed for retries, duplicates, delayed delivery, and operator review. A single successful test call is not enough for launch.

Core design

Each received event should be stored with provider event ID, received time, signature status, processing state, and a request ID. Business updates should use an idempotency key so the same event cannot create duplicate records.

Failure handling

  • Invalid signatures are rejected and logged.
  • Temporary provider or database errors move to retry with backoff.
  • Unknown payload shapes move to an exception queue for review.
  • Manual replay should be available for reviewed failures.

Ordering and state

Providers do not always deliver events in the order a business user expects. The integration should compare timestamps, current state, and allowed transitions before overwriting data.

Acceptance tests

Acceptance should include duplicate event replay, missing fields, invalid signature, rate-limit response, delayed delivery, and a manual retry from the exception queue.
